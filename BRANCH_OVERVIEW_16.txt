Branch: 17-add-access-token-and-refreshtoken-flow

====================================================
WHAT THIS BRANCH DOES
====================================================

Implements Apple Sign In authentication with a JWT access/refresh token system.
The iOS app authenticates users natively via Apple, sends the identity token to
the backend, and receives short-lived access tokens and long-lived refresh tokens
to manage sessions securely.


====================================================
FILES
====================================================

backend/app/auth/routes.py       — Auth endpoints (login, refresh, logout)
backend/app/auth/apple.py        — Apple identity token verification logic
backend/app/auth/middleware.py   — require_auth decorator for protected routes
backend/app/database/database.py — Auth DB methods (users, refresh_tokens table helpers)
backend/app/config.py            — Added JWT_SECRET and APPLE_BUNDLE_ID


====================================================
FULL AUTH FLOW
====================================================

--- LOGIN (FIRST TIME) ---

  1. iOS user taps "Sign in with Apple"
       - Apple handles the login natively on device
       - Apple returns an identity_token (a signed JWT) to the iOS app
       - Apple also returns the user's full_name on first login only
         (Apple never sends the name again after the first time)

  2. iOS calls POST /auth/login  { identity_token: "...", full_name: "..." }
       - Backend fetches Apple's current public keys from:
         https://appleid.apple.com/auth/keys
       - Matches the key used to sign the token (by kid in the header)
       - Verifies the token signature, audience (APPLE_BUNDLE_ID), and expiry
       - Extracts from the payload:
           sub   — Apple's stable unique user ID (used as apple_id in DB)
           email — present on first login only, may be a relay address
       - Looks up user in DB by apple_id
       - If user doesn't exist, creates them (stores name/email now since
         Apple won't send them again)
       - Generates:
           access_token  — JWT signed with JWT_SECRET, expires in 15 minutes
           refresh_token — random 64-byte URL-safe string, expires in 1 year
       - Stores SHA-256 hash of the refresh_token in the DB (never raw)
       - Returns: { access_token, refresh_token }


--- MAKING AUTHENTICATED API REQUESTS ---

  iOS attaches the access token to every request:
    Authorization: Bearer <access_token>

  The require_auth middleware decorator on protected routes:
    - Reads the Bearer token from the Authorization header
    - Decodes and verifies it using JWT_SECRET
    - If valid     → sets g.user_id and allows the request through
    - If expired   → returns 401 { error: token_expired }
    - If invalid   → returns 401 { error: invalid_token }


--- REFRESHING THE ACCESS TOKEN (every 15 min) ---

  iOS calls POST /auth/refresh  { refresh_token: "..." }
    - Backend hashes the raw token with SHA-256
    - Looks it up in the refresh_tokens table
    - Checks: not revoked AND not expired (1 year TTL)
    - Returns a new: { access_token }

  The refresh token is NOT rotated — same one is valid for 1 year.


--- LOGOUT ---

  iOS calls POST /auth/logout  { refresh_token: "..." }
    - Backend hashes the token and sets revoked = TRUE in the DB
    - That refresh token can never be used again
    - Returns: { ok: true }


====================================================
SECURITY DESIGN
====================================================

  Access token TTL = 15 min
    Short-lived so a stolen token expires quickly with minimal damage.

  Refresh token stored as SHA-256 hash
    A database breach does not expose usable tokens.

  Apple token verified server-side
    The identity_token is verified against Apple's live public keys,
    checking signature, audience, issuer, and expiry. Cannot be faked.

  Name/email saved on first login
    Apple only sends user info once. The backend stores it immediately
    so it isn't lost on subsequent logins.

  Apple Sign In instead of passwords
    No password storage on our end. Apple handles credential security,
    Face ID / Touch ID, and 2FA at the OS level.


====================================================
CONFIGURATION
====================================================

Environment variables required (add to .env):

  JWT_SECRET       — Secret key used to sign and verify access tokens (JWTs).
                     Generate with: python -c "import secrets; print(secrets.token_urlsafe(64))"

  APPLE_BUNDLE_ID  — Your iOS app's bundle ID (e.g. edu.uiuc.cs.hcesc.SensingApp)
                     Used to verify the audience of Apple's identity token.


====================================================
DATABASE TABLES REQUIRED
====================================================

  users
    id          SERIAL PRIMARY KEY
    apple_id    TEXT UNIQUE NOT NULL   -- Apple's stable user identifier (sub)
    email       TEXT                   -- May be a relay address, first login only
    full_name   TEXT                   -- First login only

  refresh_tokens
    id          SERIAL PRIMARY KEY
    user_id     INTEGER REFERENCES users(id)
    token_hash  TEXT UNIQUE NOT NULL   -- SHA-256 hash of the raw refresh token
    expires_at  TIMESTAMP WITH TIME ZONE
    revoked     BOOLEAN DEFAULT FALSE
